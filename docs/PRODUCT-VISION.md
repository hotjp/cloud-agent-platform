## 一、本质定位

**表面**: Agent 调度平台
**实际**: 数据飞轮 + 商业验证引擎

Cloud Agent Platform 不是单一产品，是一个底座。在它上面接不同的 GUI，就能快速验证不同的商业模式。

## 二、数据飞轮

```
用户免费使用平台调度 Agent
  ↓
产生完整任务链路（prompt → 拆解 → 执行 → 结果 → 反馈）
  ↓
这些链路本身就是高质量人工标注数据
  ↓
按来源模型分池 + 按用户质量分层 + 按场景分类
  ↓
数据清洗 → 训练数据
  ↓
训练出更好的模型
  ↓
平台能力更强 → 吸引更多用户 → 更多数据
  ↓
循环
```

### 数据维度

传统 AI 训练花钱请人标注。我们平台上天然存在：

| 数据类型 | 来源 | 训练用途 |
|----------|------|----------|
| 标注问题 | 用户 prompt | 指令理解训练 |
| 推理链 | Agent 执行过程 | Chain-of-Thought 训练 |
| 标注答案 | 任务最终结果 | 能力训练 |
| 质量评分 | 用户满意度反馈 | 偏好学习 |
| 多模型对比 | 同任务不同模型回答 | DPO/RLHF |
| 用户画像 | 使用行为 | 质量分层 |

### 用户画像分层

| 层级 | 行为特征 | 数据价值 | 处理方式 |
|------|----------|----------|----------|
| 高质量 | 复杂任务、专业领域、结构化 prompt | 🟢 极高 | 优先导出 |
| 普通 | 日常任务、中等复杂度 | 🟡 中等 | 批量清洗 |
| 低质量 | 闲聊、算命、问星座 | 🔴 低 | 过滤或安全训练 |

画像标记维度：任务复杂度、专业领域、输入质量、使用频率、满意度反馈。

### 按来源模型分池

- Claude Opus 质量高 → 优先训练种子数据
- GLM 性价比高 → 补充数据
- 新模型接入 → 独立分池
- 支持混合导出（70% Claude + 30% GLM）
- 支持对比导出（DPO/RLHF 偏好学习）

## 三、底座 + GUI 分离

```
Cloud Agent Platform（底座，造一次）
  ├── MCP 接口（唯一标准接口）
  ├── 任务调度引擎
  ├── Agent 编排
  ├── 数据采集 + 训练数据管道
  └── 多租户 + 用户画像
       ↓
  ════════ MCP 接口层 ════════
       ↓
  GUI-1: 代码助手 Web UI    → 开发者市场
  GUI-2: 数据分析 Dashboard → 企业分析市场
  GUI-3: AI 写作工具        → 创作者市场
  GUI-4: 视频生成 Studio    → 内容市场
  GUI-5: ???                → 用户自己发明
```

### 核心原则

1. **底座只造一次** — 调度、编排、数据管道全部复用
2. **GUI 是轻量的** — 只做交互层，1-2 周一个
3. **商业验证成本极低** — 新方向 = 新 GUI + 接 MCP
4. **某个 GUI 会跑通** — 不预测哪个，全部快速试
5. **底层改了 GUI 不用改** — 通过 MCP 解耦

### 对架构的硬约束

- MCP 是唯一接口，所有 GUI 通过 MCP 调用
- 接口设计足够通用，不能硬编码代码场景假设
- 任务类型开放枚举，不写死
- 数据模型支持任意场景的任务元数据
- 认证/计费/限额在底座层统一

## 四、用户画像

### 角色定义

| 角色 | 看到什么 | 交互方式 | 核心诉求 |
|------|----------|----------|----------|
| 管理员 | 全局：容器数/Agent状态/安全拦截/成本 | Web Dashboard、推送 | 一眼全局 |
| 任务提交者 | 自己的任务进度/Agent日志 | Web UI、MCP、CLI（可选） | 结果靠谱 |
| 运维 | 基础设施：资源率/Worker健康/告警 | Grafana | 系统稳定 |
| 安全审计 | 沙箱逃逸/越权/敏感访问 | 审计报告 | 合规可追溯 |

### 多租户

- 默认共享池（成本最优）
- 企业版可购专属 Agent 池
- 每个用户独立标识（tenant_id + user_id）
- 用户看自己的任务，管理员看全局

### 终端形态

Web Dashboard 是主入口，CLI 是高级选项，MCP 给 AI Agent 用，消息推送做告警，Grafana 做运维，审计报告做合规。不只有 Web UI。

### Schema 关键字段

```go
type ConversationTurn struct {
    ID           string    // ULID
    SessionID    string    // 所属 session
    TaskID       string    // 所属 task
    SubtaskID    string    // 所属 subtask
    Role         string    // user / assistant / system / tool
    Content      string    // 对话内容

```

### 导出能力

```bash
# 按模型导出
export --model claude-opus-4 --quality >= 70 --format jsonl

# 混合导出
export --mix claude-opus-4:0.7,glm-5:0.3 --format jsonl

# 对比导出（DPO/RLHF）
export --compare --models claude-opus-4,glm-5

# 增量导出
export --since 2026-05-01 --model claude-opus-4
```

- 上下文压缩前后对比 → 压缩能力训练
- 安全拦截记录 → 安全合规训练
- 多场景覆盖 → 模型能力全面化

## 八、容器生命周期调度 — 减少启停开销

> **记录时间**: 2026-05-14
> **核心理念**: 容器的成本不在运行，在启停。通过调度策略压缩容器生命周期开销。

### 问题

每次 Agent 执行都需要一个容器。Docker 容器冷启动约 200-500ms，CubeSandbox MicroVM 约 60ms，但加上环境初始化（依赖安装、代码注入）可能要 5-30 秒。高频调度场景下，启停开销会吃掉大量时间和算力。

### 容器池分层设计

```
┌──────────────────────────────────────────────────┐
│                 调度请求进来                       │
│                      ↓                            │
│   ┌─────────────────────────────────┐             │
│   │ 热池：已注入内容、暂停状态的容器   │ ← 命中：0ms 启动 │
│   │   resume → 执行 → pause         │             │
│   └─────────────┬───────────────────┘             │
│                 ↓ 未命中                           │
│   ┌─────────────────────────────────┐             │
│   │ 温池：标准化容器、环境就绪、空内容  │ ← 命中：注入内容 ~1s │
│   │   注入业务内容 → 执行 → 清空状态   │             │
│   └─────────────┬───────────────────┘             │
│                 ↓ 未命中                           │
│   ┌─────────────────────────────────┐             │
│   │ 冷池：需要从镜像创建新容器         │ ← 命中：5-30s │
│   │   create → init → inject → run  │             │
│   └─────────────────────────────────┘             │
└──────────────────────────────────────────────────┘
```

### 容器状态机

```
冷启动 → 初始化 → 空闲(温池)
                        ↓ 注入内容
                    就绪(热池)
                        ↓ resume
                    运行中
                        ↓ 任务完成
                    清空状态 / pause
                   ↙          ↘
          高频调度：pause        低频调度：清空状态
          等待下次复用           回到温池
                                    ↓
                              空闲 >10min
                                    ↓
                              销毁回收资源
```

### 三层调度策略

| 层级 | 状态 | 响应时间 | 适用场景 | 资源消耗 |
|------|------|----------|----------|----------|
| 热池 | paused，内容已注入 | ~0ms | 高频调度的同一类任务 | 内存占用，CPU 不占 |
| 温池 | 就绪，环境初始化完毕，无业务内容 | ~1s | 新任务，需要注入内容 | 内存占用 |
| 冷池 | 需要从镜像创建 | 5-30s | 温池不够时的扩容 | 创建时有 CPU 开销 |

### 资源回收策略

| 空闲时间 | 动作 | 原因 |
|----------|------|------|
| 任务完成 | pause 或 清空状态 | 高频复用 |
| 10 分钟空闲 | 销毁容器 | 释放内存和资源 |
| 温池超过上限 | LRU 淘汰最久未用 | 控制资源总量 |
| 系统负载高 | 主动缩容温池 | 保护宿主机 |

### 预热策略

- 根据历史调度模式预测下一时段的任务类型和并发量
- 提前创建标准化容器放入温池
- 比如每天 9:00 是提交高峰 → 8:50 开始预热
- 新任务类型出现 → 自动创建对应的标准化镜像

### 长期目标：FaaS 化

类似阿里云函数计算的模式：
- **急用即走**：请求来了立刻分配容器，执行完不持有
- **预留实例**：高优先级/高频率的函数保持温热
- **按需扩缩**：根据并发量自动调整容器数量
- **资源计费**：按实际执行时间计费，空闲不收费

### 对 Worker 池当前代码的要求

1. **容器复用**：Worker 执行完任务后不销毁，清空状态回温池
2. **pause/resume**：支持容器暂停和恢复（Docker pause/unpause）
3. **内容注入**：标准化容器 + 业务内容动态注入（不重新 build 镜像）
4. **空闲回收**：后台 goroutine 定期检查空闲容器，超时销毁
5. **预热机制**：配置预热规则，按时间/按类型提前准备容器
6. **弹性伸缩**：根据队列深度和并发量动态调整温池大小

## 九、未来架构优化：InProcess 执行模式

> **记录时间**: 2026-05-16
> **来源**: 创始人创始人与 AI 助手（锻）的方案讨论
> **状态**: 暂不实施，记录供未来参考

### 背景

当前架构：每个 Agent 执行 = 创建一个独立的 cap-worker Docker 容器，挂载 Git 容器的 volume，执行完毕后销毁。

随着并行 Agent 数量增长，容器启停开销（2-5s 启动延迟 + 每容器几百 MB 内存）会成为瓶颈。

### 方案：InProcess Backend

在 Git 容器内直接执行 Agent 进程（`docker exec`），不创建独立容器。

```
当前：Git 容器 + N 个 Worker 容器 → N+1 个容器
未来：Git 容器内跑 N 个 Agent 进程 → 1 个容器
```

### 架构设计

```
config: sandbox.mode = "inprocess" | "docker" | "cube" | "k8s"

                Scheduler
                    │
              Backend 接口（已有）
            ┌───────┼───────┐
            ↓       ↓       ↓
      InProcess   Docker   K8sPod
      (轻量)     (当前)   (远期)
```

**不改 Backend 接口**，新增 `InProcess` 实现：
- `Create` → 空操作（复用 Git 容器）
- `Exec` → `docker exec` 到 Git 容器执行
- `Destroy` → 空操作

### 新增组件：并发控制器

多 Agent 共享同一容器，需要协调文件访问：
- Agent 执行前声明要读写的文件范围
- 冲突时排队，不冲突时并行
- 调度层控制，不需要 AI 决策

### 后端选择策略

**不让 AI 判断，用静态规则：**

```yaml
sandbox_rules:
  default: inprocess
  override:
    - condition: "task.type == 'unsafe_code_execution'"
      backend: docker
    - condition: "task.requires_gpu == true"
      backend: k8s
    - condition: "task.duration > 30min"
      backend: docker
```

MCP 接口不暴露后端选择，`task_submit` 不变，后端选择是平台内部实现细节。

### 收益预估

| 指标 | 当前（Docker） | 未来（InProcess） |
|------|--------------|-----------------|
| 启动延迟 | 2-5s | <100ms |
| 10 Agent 内存 | ~2-3GB | ~500MB |
| 文件 I/O | 跨容器 volume | 容器内直读 |
| 容器管理复杂度 | N+1 | 1 |

### 触发条件（什么时候做）

- 并行 Agent 数量上到 50+
- Agent 任务执行频繁且短（几秒到几十秒），启动延迟成为瓶颈
- 需要快速迭代调试的场景

### 不改的原因

当前任务量下，Docker 方案的额外开销完全可接受。隔离性天然好，不用写并发控制器。Backend 接口已做好抽象，到时候加一个 InProcess 实现就行，不会返工。

### 改动量预估

- 新增：`inprocess_backend.go`（~150 行）+ `concurrency_controller.go`（~200 行）
- 改动：`config.go`（加 sandbox.mode）、`orchestrator_adapter.go`（路由后端选择）
- 不动：现有 Docker/Cube 沙箱代码

## 十、灵感收集（未整理）

> 这里记录零散的产品灵感，尚未归类。积攒到一定量后整理进上面的章节。

### Agent 能力市场
- 第三方开发者上传自定义 Agent 角色 + 工具集
- 用户按次付费调用，平台 + 开发者分成
- 优质 Agent 自然筛选，对话数据质量更高 → 训练价值更大
- 渐进策略：PGC → PUGC（邀请制）→ UGC（严格审核），参考扣子砍掉 UGC 的教训（内容安全+质量失控）

### 任务模板即产品
- 高频成功的任务执行链路固化为模板
- 模板 = 编排链路 + 上下文策略 + 角色分配的最佳实践
- 极简 GUI：一个输入框 + 一个按钮 = 一个独立产品
- 模板积累越多 → 商业验证越快

### 容器调度优化灵感
- 容器成本在启停不在运行
- 高频调度：pause 不销毁，下次 resume（0ms）
- 低频调度：清空状态回温池（~1s）
- 10 分钟空闲：销毁回收
- 长期目标：FaaS 化，急用即走，预留实例保温，按执行时间计费
- 预热策略：根据历史模式预测并发，提前准备温池容器
