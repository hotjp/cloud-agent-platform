## 一、文件访问延迟测试

### 目标
找到文件操作从"可接受"到"不可接受"的临界点。

### 测试矩阵

| 编号 | 测试场景 | 测量指标 | 极限值 |
|------|---------|---------|--------|
| F-1 | 单次 read_file（1KB 文件） | P50/P95/P99 延迟 | — |
| F-2 | 单次 read_file（10MB 文件） | P50/P95/P99 延迟 | — |
| F-3 | 连续 read_file 100 次（模拟 Agent 读项目） | 总耗时 / 单次平均 | 总耗时 < 5s |
| F-4 | 连续 read_file 1000 次 | 总耗时 / 单次平均 | 找到退化点 |
| F-5 | write_file 10MB 文件 | 延迟 | — |
| F-6 | 10 个 Agent 同时 read 同一文件 | P99 延迟 vs 单 Agent | 延迟增幅 < 50% |
| F-7 | 10 个 Agent 同时 read 不同文件 | P99 延迟 | — |
| F-8 | 1 个 Agent 连续 write 100 个文件 | 总耗时 | — |
| F-9 | 10 个 Agent 同时 write 同一文件（无锁） | 冲突率 / 数据损坏率 | 应 100% 有冲突 |
| F-10 | 10 个 Agent 同时 write 同一文件（有文件锁） | 冲突解决成功率 / 等待时间 | 100% 成功 |
| F-11 | 1 个 Agent write + 另 1 个 Agent 同时 read 同一文件 | 读到脏数据的概率 | 应为 0 |
| F-12 | 批量 batch_read 50 个文件 vs 单次 50 次 read | 总耗时对比 | batch 应快 3x+ |

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| F-X1 | 单文件 1GB，read_file | 找到单文件大小上限 |
| F-X2 | 10 个 Agent 同时 write 同一文件，持续 60 秒 | 文件锁饥饿 / 死锁 |
| F-X3 | 50 个 Agent 同时操作同一项目 | 找到并发崩溃点 |
| F-X4 | Volume 磁盘写满时 write_file | 错误处理 |
| F-X5 | Git 容器重启后 Agent 的文件操作 | 恢复能力 |

### 测试方法

```bash
# 自动化测试脚本（伪代码）
for test in F-1..F-X5:
  setup_test_env()
  for i in 1..100:  # 跑 100 次取统计值
    start_timer()
    result = agent.call_tool(test.tool, test.params)
    stop_timer()
    record(result, latency)
  report_p50_p95_p99()
  teardown_test_env()
```

## 二、上下文传递成本测试

### 目标
找到初始上下文的合理大小——太小 Agent 理解不了项目，太大成本不可接受。

### 测试矩阵

| 编号 | 上下文大小 | 包含内容 | 测量指标 |
|------|-----------|---------|---------|
| C-1 | 0 tokens（只有任务描述） | — | Agent 能否完成任务？需要多少轮 read_file？ |
| C-2 | ~500 tokens | 文件树（文件名+目录结构） | Agent 能否定位到目标文件？ |
| C-3 | ~2K tokens | 文件树 + README.md | — |
| C-4 | ~10K tokens | 文件树 + README + 主要源文件摘要 | — |
| C-5 | ~50K tokens | 文件树 + 所有源文件 | — |
| C-6 | ~200K tokens | 整个项目全量 | LLM 调用成本（元） |

### 测量方法

对每个上下文大小，给 Agent 一个标准任务（如"修复 main.go 中的空指针错误"），记录：

1. **任务完成率** — 10 次中成功几次
2. **额外 read_file 次数** — Agent 需要读多少额外文件才能找到目标
3. **LLM 调用总 token 数** — 包含上下文 + 后续交互
4. **总成本** — 按 GLM-5 定价估算
5. **总耗时** — 从任务开始到完成

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| C-X1 | 上下文 500K tokens | LLM 网关是否扛得住？延迟多少？ |
| C-X2 | 上下文 1MB tokens | 超出模型上下文窗口时如何降级？ |
| C-X3 | 10 个 Agent 同时拉取 50K 上下文 | 平台并发吐上下文的能力 |
| C-X4 | 上下文包含二进制文件（图片/压缩包） | 过滤和降级策略 |

## 三、并发控制极限测试

### 目标
找到文件锁从"正常工作"到"性能崩溃"或"死锁"的边界。

### 测试矩阵

| 编号 | 场景 | 测量指标 |
|------|------|---------|
| L-1 | 2 个 Agent 写不同文件（无冲突） | 延迟 vs 单 Agent |
| L-2 | 2 个 Agent 写同一文件（悲观锁） | 等待时间 |
| L-3 | 5 个 Agent 写同一文件 | 等待时间分布 |
| L-4 | 10 个 Agent 写同一文件 | 等待时间分布 |
| L-5 | 20 个 Agent 写同一文件 | 等待时间 / 有无死锁 |
| L-6 | Agent A 写文件 X，Agent B 写文件 Y，Agent C 写 X+Y | 跨文件锁 |
| L-7 | Agent 持锁后崩溃（不释放） | 锁自动释放时间 |
| L-8 | Agent 持锁 60 秒（超长操作） | 锁超时机制是否生效 |
| L-9 | 乐观锁：2 Agent 同时写，第二个提交时检测冲突 | 冲突检测成功率 |
| L-10 | 10 Agent 并发 git commit（平台统一 commit） | commit 冲突率 |

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| L-X1 | 50 个 Agent 同时写同一项目的不同文件 | 找到并发写入吞吐量上限 |
| L-X2 | 50 个 Agent 同时写同一文件 | 死锁 / 饥饿 / 性能崩溃 |
| L-X3 | Agent 嵌套锁（锁文件 A → 锁文件 B → 释放在相反顺序） | 死锁检测 |
| L-X4 | 每秒 100 次锁获取/释放 | 锁管理器的吞吐量上限 |
| L-X5 | 10 个 Agent 各写 100 文件后同时触发 git commit | 找到 Git 操作的并发上限 |
| L-X6 | Git 容器在做 git push 时，50 个 Agent 同时 write_file | push 期间文件操作是否阻塞 |

## 四、Agent 故障恢复测试

### 目标
验证 Agent 崩溃后任务状态能正确恢复，找到恢复机制的边界。

### 测试矩阵

| 编号 | 场景 | 预期结果 |
|------|------|---------|
| R-1 | Agent 正常运行中 kill -9 | 心跳超时（90s）→ 租约过期 → 锁释放 → 任务回 pending |
| R-2 | Agent 写了 50% 文件后 kill -9 | 已写文件是否需要回滚？git checkout -- . 是否正确 |
| R-3 | Agent 持有 3 个文件锁时 kill -9 | 3 个锁全部自动释放 |
| R-4 | 10 个 Agent 中 kill 其中 3 个 | 其他 7 个不受影响，被 kill 的正确恢复 |
| R-5 | Agent 卡死（不崩溃但不发心跳） | 90s 后判定失联 → 触发恢复 |
| R-6 | Agent 网络断开（与平台失去连接） | 同 R-5 |
| R-7 | 恢复后任务重新分配给新 Agent | 新 Agent 能正确接手 |

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| R-X1 | 50 个 Agent 同时崩溃 | 平台恢复能力 |
| R-X2 | 同一任务连续崩溃 5 次 | 是否有重试上限？是否拆分为更小的子任务？ |
| R-X3 | Agent 崩溃 → 恢复 → 再崩溃 → 再恢复，循环 10 次 | 系统状态是否一致？有无资源泄漏？ |
| R-X4 | Agent 崩溃时正在做 git commit | commit 状态：原子性是否保证？ |
| R-X5 | Agent 崩溃时 LLM 网关正在返回结果 | 网关如何处理悬空连接？ |

## 五、LLM 网关极限测试

### 目标
找到 LLM 网关的吞吐量上限和降级边界。

### 测试矩阵

| 编号 | 场景 | 测量指标 |
|------|------|---------|
| G-1 | 单 Agent ask_llm（1000 tokens prompt） | 延迟 / 成本 |
| G-2 | 单 Agent ask_llm（100K tokens prompt） | 延迟 / 成本 |
| G-3 | 5 Agent 同时 ask_llm | 延迟增幅 |
| G-4 | 20 Agent 同时 ask_llm | 延迟增幅 / 有无限流 |
| G-5 | 单 Agent 连续 ask_llm 100 次 | 累计成本 / 延迟变化 |
| G-6 | ask_llm 返回后检查记录的 input/output | 数据完整性 |

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| G-X1 | 100 Agent 同时 ask_llm | 网关是否崩溃？限流策略是否生效？ |
| G-X2 | ask_llm prompt 1MB | 网关内存 / 处理能力 |
| G-X3 | LLM Provider 返回 500 错误 | 网关重试 / 降级 / 错误传递 |
| G-X4 | LLM Provider 超时 60 秒 | 网关超时策略 |
| G-X5 | 网关缓存命中测试（相同 prompt 第二次调用） | 缓存有效性 / 成本节省 |
| G-X6 | 单任务 ask_llm 次数超过 10 次上限 | 限流是否正确拦截 |

## 六、端到端极限测试

### 目标
全链路压测，找到整个系统的崩溃点。

### 测试矩阵

| 编号 | 场景 | 测量指标 |
|------|------|---------|
| E-1 | 1 个任务 → 1 个 Agent → 完成 | 基线：总耗时 / 成本 / 正确性 |
| E-2 | 1 个大任务 → 拆解为 5 个子任务 → 5 Agent 并行 | 加速比 vs 串行 |
| E-3 | 5 个任务 → 10 Agent 并行 | 系统吞吐量 |
| E-4 | 10 个任务 → 20 Agent 并行 | 吞吐量 / 资源使用 |
| E-5 | 任务 A 依赖任务 B 的结果 | 依赖调度正确性 |

### 极限测试

| 编号 | 极限场景 | 目的 |
|------|---------|------|
| E-X1 | 100 个任务同时提交 | 任务队列 / 调度器崩溃点 |
| E-X2 | 1 个任务拆解为 50 个子任务，50 Agent 并行 | 并行度上限 |
| E-X3 | 连续 24 小时，每分钟提交 5 个任务 | 长时间运行稳定性 / 内存泄漏 |
| E-X4 | 所有任务操作同一个项目 | 最极端的并发冲突 |
| E-X5 | 提交任务后立即取消 → 再提交 → 再取消（循环 100 次） | 状态机正确性 |
